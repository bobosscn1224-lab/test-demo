# 数字分身 — 项目宪法

> 所有开发、AI 调用、代码变更必须遵循本文档。改动先更新文档。

---

## 一、核心设计原则

### 0. 需求驱动开发流程（铁律）

**拿到需求后，禁止直接动手改代码。必须先完成以下步骤：**

```
需求 → 分析 → 设计 → 评审 → 实现 → 验证
```

| 步骤 | 必须做的事 | 禁止跳过 |
|------|-----------|---------|
| **分析** | 读当前架构、数据流、已有的规范文件，理解现状 | ❌ 不看代码就提方案 |
| **设计** | 基于现有设计模式（持久化规范、共享模块、v1/v2 共存）提出方案，说明改动范围和影响 | ❌ 不考虑架构一致性 |
| **评审** | 确认方案与项目宪法一致，复用已有规范而非新建 | ❌ 方案未确认就开始写 |
| **实现** | 按方案写代码，先改规范文件再改代码 | ❌ 边写边改方向 |
| **验证** | 跑编译、启动服务、端到端测试 | ❌ 不测试就说完成 |

**判断标准**：如果改动后需要把同样的规则在多个地方各改一遍 → 设计有问题，应该抽取共享模块。

### 1. 大模型交互标准范式（铁律）

**每一次与大模型（LLM / Image Model）的交互，必须严格遵循以下范式：**

```
┌──────────────────────────────────────────────────────────────┐
│                    STANDARD LLM INTERACTION                    │
│                                                               │
│  1. SPEC   → 从持久化规范文件加载：要求、规则、输入            │
│  2. CALL   → 调用大模型，传入规范化的 prompt                   │
│  3. CHECK  → 校验输出是否满足质量要求                          │
│  4. RETRY  → 不满足？用具体反馈让大模型完善（最多 N 次）        │
│  5. GATE   → 只返回通过所有质量检查的结果                      │
└──────────────────────────────────────────────────────────────┘
```

**禁止行为**：
- ❌ 直接调用 LLM 而不经过质量校验
- ❌ 在业务代码中硬编码 prompt 规则
- ❌ 接受不满足要求的输出
- ❌ 跳过重试机制

**实现方式**：
- 所有 LLM 交互使用 `app/services/llm_interaction.py` 的 `execute_with_quality_gate()`
- 所有图像生成使用 `app/services/collage_prompt_spec.py` 的 `build_collage_prompt()` + `validate_output()`
- 交互规范持久化在 `app/services/llm_interaction_spec.yaml`
- 图像生成规范持久化在 `app/services/collage_prompt_spec.yaml`

### 2. 持续改进与问题回溯原则（铁律）

**每次出现问题或 Bug，必须完成以下闭环，不得只修不究：**

```
发现问题 → 定位根因 → 记录教训 → 完善规范 → 交付
```

| 步骤 | 必须做的事 |
|------|-----------|
| **定位根因** | 不是修表面现象，是找到造成问题的**关键缺失点**。例如：是 prompt 没约束？是校验没做？是规范文件没更新？ |
| **记录教训** | 将根因和教训写入 `AGENTS.md` 的「问题回溯记录」章节，记录：什么现象、根因是什么、哪个环节没做好、如何防止复发 |
| **完善规范** | 根据教训更新对应的规范 YAML 文件或设计原则，确保同类问题不会复现 |
| **验证闭环** | 修复后验证，确认问题解决 + 规范已更新 |

**禁止行为**：
- ❌ 只修 bug 不分析根因
- ❌ 同样的错误出现两次
- ❌ 发现问题是因为规范缺失，却不补规范

### 3. 持久化优先原则

**任何规则、约束、prompt 模板必须持久化到规范文件中，不得硬编码。**

| 持久化文件 | 管理内容 |
|-----------|---------|
| `app/services/llm_interaction_spec.yaml` | LLM 交互的质量检查规则、重试模板、模型参数 |
| `app/services/collage_prompt_spec.yaml` | 拼图生图的所有规则：网格约束、视觉方向、内容处理 |
| `DEVELOPMENT_CONTRACT.md` | 前后端接口契约、字段映射、状态机 |
| `AGENTS.md` | 本文档 — 项目级设计原则和开发规范 |

### 4. 单一真相来源原则

**每条规则只有一个权威定义位置。** 如果一条规则出现在两个地方，这是 bug。

## 二、问题回溯记录

> 每次出现问题，记录：现象、根因、哪个环节没做好、如何防止复发。

### 2026-07-17：视觉模型故障时误放行几何不确定的拼图

| 项 | 内容 |
|----|------|
| **现象** | 真实并发生图中 A/C 无法分割网格，Tutujin `gemini-3-pro-thinking` 同时返回 503，系统将其标记为 `passed_with_manual_review`；人工检查发现末行高度实际不一致 |
| **根因** | `on_detection_failure: vision_review` 把硬性几何不变量交给了可失效的远程视觉模型；视觉失败降级没有区分“几何已通过”与“几何未知” |
| **缺失环节** | 未用真实付费调用验证“几何检测失败 + Vision 503”的组合故障 |
| **修复** | 网格分割不确定改为硬失败并定向重试；`passed_with_manual_review` 只能用于本地几何已通过、语义审查不可用的候选图 |
| **防止复发** | 所有硬性尺寸、数量、页序不变量必须由本地可重复检查确认；远程视觉模型不得替代几何真值源 |

### 2026-07-17：Tutujin 模型列表与实际通道权限不一致

| 项 | 内容 |
|----|------|
| **现象** | Vision Key 的 `/v1/models` 返回 `gemini-3-pro-thinking`，但真实调用返回 `model_not_found / No available channel`；生图 Key 列出多个 Gemini，但调用 2.5 Flash 时返回无权访问另一个 3.1 模型 |
| **根因** | 供应商的模型目录、分组通道和实际路由授权漂移，“列表中存在”不等于“可调用” |
| **修复** | 备用模型顺序持久化并支持自动降级，但生产配置在真实通道修复前保持空列表，避免无效调用和延时 |
| **防止复发** | 视觉模型上线必须做真实最小文本调用和带图调用，不得只依赖 `/v1/models` 列表判定可用性 |

### 2026-07-17：Tutujin VIP 并发生成长时间空响应

| 项 | 内容 |
|----|------|
| **现象** | 真实付费 A/B/C 批次耗时 720.45 秒；A 空响应，B 两次空响应，C 在排队后超时，无图可进入视觉评估。 |
| **根因** | Tutujin VIP 在并发调用下多次返回 HTTP 200 + 空 content；单图超时未覆盖信号量排队；调用失败和质量失败没有共享付费调用预算；前端无真实阶段进度。 |
| **缺失环节** | 未用真实付费结果验证供应商并发能力、整批最坏耗时、重试费用上限和进度可见性。 |
| **修复** | PPT 拼图固定 Tutujin VIP 单通道 A→B→C；单次调用 300 秒，整批 900 秒；每方案最多两次付费调用；增加持久化进度和完整批次校验。 |
| **防止复发** | 供应商并发数、单次/整批超时、冷却和付费次数必须在 YAML 中定义；上线前必须执行一次真实付费验收并记录每张耗时与门禁结果。 |

### 2026-07-17：正确 3×4 拼图被整图 16:9 和归一化门禁误拒

| 项 | 内容 |
|----|------|
| **现象** | 真实生成的 10 页拼图中，每个缩略页等尺寸且为 16:9，但整图为 4:3，被门禁以“实际 1.333，期望 1.750”拒绝；无边框卡片检测到 0 个并再次硬拒。 |
| **根因** | 将“单个缩略页 16:9”错误套用为“整张拼图 16:9”。10 页按 3×4 排列时，整图自然比例应为 `(3×16/9)/4 = 4/3`；低置信度分割器又被他处改为强制归一化和硬拒。 |
| **缺失环节** | 没有用真实拼图验证“整图比例”的数学定义；规范中“分割不确定交给语义门禁”被后续修改覆盖。 |
| **修复** | 整图期望比例按 `(列数×单格比例)/行数` 动态计算；关闭自动归一化硬门禁；分割不确定时必须由视觉语义门禁检查页数、等尺寸和 16:9。 |
| **防止复发** | 整图画布比例必须由页数/行列数动态推导，不得直接使用单页比例；任何硬几何门禁都必须通过真实样本回归。 |

### 2026-07-16：PPT 风格方案长时间无响应且 A/B/C 批次混合

| 项 | 内容 |
|----|------|
| **现象** | 三套方案串行生成耗时过长，中途失败后项目保留新 A + 旧 B/C；重生也可能覆盖有效图。 |
| **根因** | 前端将一个业务批次拆为三个单图请求，后端逐张合并持久化；包装层未对生图协程应用真实超时；供应商重试与质量门禁重试叠加。 |
| **缺失环节** | 缺少 run 级原子性、最坏耗时计算和“一个失败不得改变当前批次”的回归测试。 |
| **修复** | A/B/C 使用同一 `run_id` 在临时批次生成，全部通过后才一次写回；真实协程超时；只保留一层质量重试。 |
| **防止复发** | 多产物生成必须有批次 ID 和原子提交；业务超时必须直接约束最外层协程；禁止不同层同时重试同一付费调用。 |

### 2026-07-16：真实 PPT 拼图被合成图几何算法误判

| 项 | 内容 |
|----|------|
| **现象** | 真实生成图视觉上有完整页面，但连通域检测只找到 1 个区域并拒绝交付。 |
| **根因** | 检测器假设每个缩略页与画布存在明显色差边界，但无边框、深色和通栏设计不满足该假设；测试只使用了纯色矩形。 |
| **缺失环节** | 没有用历史真实 gpt-image 输出建立阳性/阴性回归集，将低置信度的诊断算法当成硬门禁。 |
| **修复** | 文件与画布检查保持硬门禁；分割算法仅输出诊断；页数、页序、同尺寸和末行不拉伸由专用视觉语义门禁强制。 |
| **防止复发** | 图像几何硬门禁上线前必须通过真实样本回归集；无法可靠分割时必须显式降级至语义门禁，不得猜测失败。 |

### 2026-07-16：PPT 拼图重试后被视觉审查模板误判失败

| 项 | 内容 |
|----|------|
| **现象** | Tutujin VIP 首次生成 3:2 拼图被画布比例门禁拒绝；重试已生成合格画布，仍报 `KeyError: '"passed"'`；仅靠 Prompt 无法证明每个缩略图尺寸一致 |
| **根因** | 视觉审查 YAML 中的 JSON 示例花括号未转义，被 `str.format()` 误认为字段；生成后质量门禁缺少缩略图网格几何校验 |
| **缺失环节** | (1) 持久化模板未做全量渲染自检 (2) 只检查整图宽高比，未检查单格宽高 (3) 前端丢弃后端质量失败原因 |
| **修复** | 转义所有审查 JSON 字面量并增加渲染回归；强化持久化 Prompt 同尺寸规则；增加生成后网格几何检查和具体错误展示 |
| **防止复发** | 所有新增 `vision_review_prompt` 必须通过模板渲染测试；PPT 拼图必须同时通过 Prompt、整图画布、单格几何和语义审查，不得仅凭 Prompt 约束放行 |

### 2026-07-16：架构基线审计发现构建、契约与运行状态漂移

| 项 | 内容 |
|----|------|
| **现象** | 前端 TypeScript 构建失败；PPT 项目的 `image_backend`、`outline_mode` 与拼图运行态字段前后端不一致；知识库扫描状态 JSON 损坏会在启动时报警；启动脚本同时使用 8001/8002/8011；已有回归测试与当前持久化模型规范矛盾 |
| **根因** | 功能快速迭代时没有把“契约更新 → 编译 → 回归测试”作为同一个原子交付；JSON 状态直接覆盖写入，进程中断时可能留下半文件；端口在多个脚本和业务代码中硬编码 |
| **缺失环节** | (1) 前后端字段变更后未运行严格 TypeScript 构建 (2) 状态文件缺少原子写入和损坏恢复 (3) 启动配置未使用单一真相来源 (4) 回归测试未随模型规范更新 |
| **修复** | 更新 `DEVELOPMENT_CONTRACT.md`；同步前后端类型；状态文件改为稳定绝对路径 + 原子写入 + 损坏隔离；统一默认端口为 8001；更新与当前规范冲突的测试 |
| **防止复发** | 每次交付必须通过：前端 `npm run build`、后端全模块导入、核心 pytest、API 冒烟；持久化 JSON 一律使用原子替换；端口只从 `Settings.port` 或启动参数读取 |

### 2026-07-16：文件接口路径边界不完整

| 项 | 内容 |
|----|------|
| **现象** | 技能上传沿用客户端文件名，下载与 PPT 项目文件路径使用字符串拼接/前缀判断，存在目录逃逸或同名前缀误判风险 |
| **根因** | 文件名清理和目录包含校验分散在多个接口中，没有共享的边界规则 |
| **缺失环节** | 接口设计与代码审查没有覆盖恶意文件名、URL 编码路径和非法项目 ID |
| **修复** | 上传文件使用 basename + 随机前缀；下载使用 `Path.resolve()` + `relative_to()` 判断；项目 ID 限制为 8 位十六进制 |
| **防止复发** | 所有新增文件接口必须验证扩展名/文件名，并使用真实路径包含判断；不得用字符串 `startswith()` 判断目录归属 |

### 2026-07-16：大知识库统计触发 SQLite 参数上限

| 项 | 内容 |
|----|------|
| **现象** | 知识库已有约 4.95 万个分块，`/api/knowledge/stats` 返回 `status=error`，但 Chroma collection 本身可正常 `count()` |
| **根因** | 统计逻辑使用一次无分页的 `collection.get(include=["metadatas"])`，Chroma 在底层生成的 SQL 超过 SQLite 变量上限 |
| **缺失环节** | 测试只覆盖小数据集，没有用现有真实索引规模验证统计接口 |
| **修复** | 总数使用 `count()`，元数据按固定批次分页读取并累计唯一文档数；异常记录具体日志 |
| **防止复发** | Chroma/SQLite 列表与统计操作必须分页，禁止对未知规模集合执行无上限 `get()` |

### 2026-07-16：周报索引中的绝对路径跨环境失效

| 项 | 内容 |
|----|------|
| **现象** | 周报记录可以列出，但旧索引保存 Windows 绝对路径，迁移到 macOS 后 `download_url` 为空 |

| **防止复发** | 持久化记录保存可迁移的相对标识；读取历史绝对路径时必须提供规范目录回退 |

### 2026-07-16：聊天页辅助文本出现乱码

| 项 | 内容 |
|----|------|
| **现象** | 浏览器无障碍树中，回答模式开关标题显示为 `鍒囨崲鍥炵瓟妯″紡` |
| **根因** | 单个中文字符串曾被用错误编码解码后写回源码，TypeScript 编译不会识别语义乱码 |

### 2026-07-16：会话标题可能暴露敏感文本且列表无上限

| 项 | 内容 |
|----|------|
| **现象** | 会话标题直接截取首条用户消息，历史标题可能包含 secret/token 等敏感片段；`GET /api/sessions` 一次返回全部历史会话，页面与响应持续膨胀 |
| **根因** | 自动标题没有经过敏感信息脱敏；列表查询没有 limit 边界 |
| **缺失环节** | 隐私审查只关注模型输出，没有检查导航标题和批量列表接口 |
| **修复** | 增加共享文本脱敏工具；新建、更新、自动生成和读取会话标题均脱敏；会话列表增加 1-200 的分页参数并默认返回最近 100 条 |
| **防止复发** | 任何由用户原文派生的摘要、标题、日志标签都必须先脱敏；所有历史列表接口必须有上限 |

### 2026-07-16：Skill 会话重启后丢失正文

| 项 | 内容 |
|----|------|
| **现象** | 飞书文档、飞书妙记等 Skill 声称会话可跨重启恢复，但数据库只保存正文长度，重启后无法继续基于原文回答 |
| **根因** | 会话持久化层为了缩小记录体积，统一排除了 `content` 字段，却没有把正文保存到其他持久化位置 |
| **缺失环节** | 持久化设计只验证了 stage/title 恢复，没有验证恢复后的核心业务数据是否完整 |
| **修复** | 共享 `SkillSessionHelper` 与飞书妙记会话保存完整数据，并保留 `content_length` 便于诊断 |
| **防止复发** | 所有声称“跨重启恢复”的会话必须用回归测试验证核心上下文，而不只验证状态字段 |

### 2026-07-16：默认测试命令误执行外部集成脚本

| 项 | 内容 |
|----|------|
| **现象** | 在 `backend` 运行 `pytest` 时会收集根目录历史脚本，并在收集阶段直接访问本地服务或外部模型接口 |

| **防止复发** | 可重复的自动测试只放入 `backend/tests`；需要服务、网络或计费接口的脚本不得在导入阶段执行 |

## 三、LLM 调用规范

### 标准调用代码

```python
from app.services.llm_interaction import execute_with_quality_gate

result = await execute_with_quality_gate(
    interaction_name="outline_generation",  # 对应 llm_interaction_spec.yaml 中的 key
    system_prompt=system_prompt,
    user_prompt=user_prompt,
    llm_service=llm_service,
    extra_context={                         # 用于重试 prompt 构建
        "source_text": source_text,
        "briefing_text": briefing_text,
    },
)

if not result.success:
    # 质量检查未通过，result.error 包含失败原因
    # result.quality_failures 包含每项失败的具体反馈
    raise Exception(f"Quality gate failed: {result.error}")
```

### 图像生成调用规范

```python
from app.services.collage_prompt_spec import (
    build_collage_prompt, strip_visual_suggestions,
    validate_prompt,
)

# 1. 清理大纲（去掉视觉建议，保留所有内容）
cleaned_outline = strip_visual_suggestions(outline)

# 2. 构建标准化 prompt
prompt = build_collage_prompt(
    total_pages=page_count,
    cleaned_outline=cleaned_outline,
    variant_label="A",
    project_context="应用场景：业务汇报\n目标受众：管理层",
)

# 3. 发送前校验
warnings = validate_prompt(prompt, page_count, cleaned_outline)

# 4. 调用统一生图边界；边界内部自动执行生成前检查、
#    临时文件隔离、生成后文件/比例/视觉复核、有限重试和原子放行
error = await image_gen.generate(
    prompt,
    output_path,
    interaction_name="ppt_collage",
    validation_context={
        "expected_pages": page_count,
        "columns": 3,
        "outline": cleaned_outline,
    },
    timeout=420,
)
```

### 新增 LLM 交互的流程

1. 在 `llm_interaction_spec.yaml` 添加新的 interaction section，定义 `quality_checks`、`retry_prompt_template`
2. 实现时调用 `execute_with_quality_gate(interaction_name="新交互名称", ...)`
3. 框架自动处理校验、重试、把关

- **共享模块**：两个版本通过 `collage_prompt_spec.py` 和 `llm_interaction.py` 共享规则

## 五、开发规范

### 4.1 修改规则流程

1. 修改对应的规范 YAML 文件
2. 如果改了字段名/接口，同步更新 `DEVELOPMENT_CONTRACT.md`
3. 写代码引用规范文件中的规则，不要复制粘贴
4. 重启后端，验证

### 4.2 Prompt 编写原则

- prompt 规则放 YAML，不放代码
- 约束用醒目格式（`━━━` 框、`MANDATORY`、`VIOLATION MEANS REGENERATION`）
- 给大模型的指令要具体：`Slide 1 = Slide 7 = Slide N in dimensions` 比 `统一尺寸` 有效
- 去掉冗余信息：传给生图模型的是**内容**，不是 LLM 臆想的**布局指令**

### 4.3 代码审查检查点

- [ ] 新 LLM 调用用了 `execute_with_quality_gate()` 吗？
- [ ] 新图像生成用了 `build_collage_prompt()` + `validate_output()` 吗？
- [ ] 有没有硬编码的 prompt 规则？（有 = 不合格）
- [ ] 质量检查的 `fail_feedback` 是否具体、可执行？
- [ ] `max_retries` 设置合理吗？
