# 全局强制执行规则

## 需求驱动开发流程（最高优先级）
**拿到需求后，禁止直接动手改代码。必须先完成以下步骤：**

```
需求 → 分析 → 设计 → 评审 → 实现 → 验证
```

1. **分析**：读懂当前架构、数据流、已有规范，理解为什么现有代码这样写
2. **设计**：基于现有设计模式提出方案，明确改动范围和影响面
3. **评审**：确认方案与项目宪法（CLAUDE.md）一致，能复用就不新建
4. **实现**：先改规范文件，再改代码；让规范驱动代码，而非代码里写死规则
5. **验证**：编译通过、服务启动、端到端测试通过

**硬性约束**：如果同样的规则需要在多个地方改 → 抽取共享模块。不要重复自己。

## 持续改进与问题回溯
**每次出现问题或 Bug，必须完成闭环，不得只修不究：**

1. **定位根因**：不是修表面现象，找到造成问题的关键缺失点（prompt 没约束？校验没做？规范没更新？）
2. **记录教训**：将根因和教训写入 `CLAUDE.md` 的「问题回溯记录」章节
3. **完善规范**：更新对应的 YAML 规范文件或设计原则，防止同类问题复发
4. **验证闭环**：确认问题解决 + 规范已更新

**硬性约束**：同样的错误不允许出现两次。发现规范缺失必须补规范。

## 大模型交互铁律
**每一次与大模型（LLM / Image Model）的交互，必须严格遵循以下范式：**

1. **SPEC** → 从持久化规范文件加载要求、规则、输入
2. **CALL** → 调用大模型，传入规范化的 prompt
3. **CHECK** → 校验输出是否满足质量要求
4. **RETRY** → 不满足？用具体反馈让大模型完善（最多 N 次）
5. **GATE** → 只返回通过所有质量检查的结果

**硬性要求**：
- 规范文件：`app/services/llm_interaction_spec.yaml` 和 `app/services/collage_prompt_spec.yaml`
- 代码入口：`app/services/llm_interaction.py` 的 `execute_with_quality_gate()`
- 禁止：硬编码 prompt 规则、跳过质量校验、接受不合格输出
- 详见：`CLAUDE.md` 项目宪法

## 核心协作体系
严格启用8角色多Agent研发流程，按固定顺序推进，禁止跳步、跨角色操作
流程顺序：产品需求分析→系统架构设计→数据库建模→后端开发→前端开发→测试校验→代码安全质检→项目汇总交付

## 各角色执行标准
1.产品分析师：拆解需求，输出结构化PRD，明确场景、边界、优先级
2.架构师：敲定技术栈、分层架构、模块边界、接口规范
3.DBA：设计数据表、索引、约束，产出可用建表语句
4.后端开发：遵循规范编码，完备异常、权限、日志处理
5.前端开发：组件化开发，适配多状态，交互体验合规
6.测试工程师：编写全量用例，排查边界与逻辑漏洞
7.质检审计：代码评审、漏洞排查、性能优化，达标上线标准
8.项目管控：统筹进度，整合资料，输出完整交付产物

## 硬性约束
1.每阶段完成审核通过后，才可进入下一阶段
2.所有产出满足可维护、可扩展、可部署、生产可用要求
3.禁止随意删减流程环节，拒绝半成品输出
4.全程遵循代码规范与安全准则
